# Shortcut UX 정리 및 판단 계획

## 문서 목적

- 현재 `skimd`의 단축키와 패널 명칭이 앱의 핵심 컨셉에 맞는지 평가한다.
- 각 기능의 정의, 사용 방법, 사용자 가치, 혼란 비용을 정리한다.
- 무엇을 유지하고 무엇을 제거하거나 숨길지 판단할 수 있는 기준 문서를 만든다.

이 문서는 구현 문서가 아니라, 단축키와 용어를 정리하기 위한 의사결정 문서다.

## 제품 컨셉 재확인

`skimd`의 핵심 사용 흐름은 다음과 같다.

1. 작업 중인 디렉토리에서 tmux popup으로 빠르게 연다.
2. 폴더를 따라가며 문서를 찾는다.
3. 문서를 열어 짧게 검토하거나 길면 조금 더 읽는다.
4. 확인이 끝나면 바로 닫고 원래 세션으로 돌아간다.

이 흐름에서 중요한 것은 다음 네 가지다.

- 처음 봐도 이해되는 용어
- 현재 화면에서 실제로 쓸 수 있는 키만 보이는 것
- 빠른 탐색과 빠른 종료
- 문서 리뷰에 직접 기여하는 기능만 남기는 것

## 현재 문제 요약

### 1. 패널 이름이 역할을 설명하지 못함

- 좌측 패널이 `Review`로 표시되지만 실제 역할은 파일/폴더 브라우저다.
- 사용자는 "왜 폴더 리스트가 Review인가?"를 자연스럽게 느끼게 된다.

### 2. 도움말이 문맥을 반영하지 못함

- browser 상태에서도 `/ search`가 보이지만 실제로는 reader에서만 동작한다.
- `[/] heading`, `a`, `o` 같은 항목은 의미를 모르면 거의 해석이 안 된다.

### 3. 고급 기능과 기본 기능이 같은 레벨로 노출됨

- `Enter`, `q`, `j/k`처럼 핵심 키와
- `a`, `o`, `z`, `[` / `]`, `r`처럼 보조 키가
- 같은 밀도로 노출되어 초반 학습 비용이 올라간다.

## 판단 기준

각 기능은 아래 기준으로 평가한다.

### A. 핵심 흐름 기여도

- popup 리뷰 흐름에서 자주 쓰이는가
- 문서 탐색이나 검토 속도를 직접 올리는가

### B. 직관성

- 이름과 키만 보고 역할을 추측할 수 있는가
- 최초 사용자가 설명 없이도 시도할 수 있는가

### C. 문맥 적합성

- 현재 화면에서 실제로 동작하는가
- 도움말에 보여줘도 오해가 없는가

### D. 대체 가능성

- 없어도 다른 키나 동작으로 충분히 대체되는가
- 유지하더라도 기본 help에서 숨길 수 있는가

## 용어 정리 후보

### 좌측 패널 `Review`

현재 의미:

- markdown 중심으로 필터링된 파일/폴더 브라우저

문제:

- 행동이나 상태를 설명하지 못한다.
- 사용자는 "리뷰 모드"인지 "브라우저"인지 구분하기 어렵다.

후보:

- `Docs`
  - markdown 위주라는 점은 잘 드러난다.
  - 디렉토리도 보이므로 완전히 정확하진 않다.
- `Files`
  - 가장 직관적이다.
  - 기본 필터가 markdown 중심이라는 점은 약해진다.
- `Browser`
  - 역할 설명은 맞다.
  - 터미널 앱 용어로는 약간 딱딱하다.

권장:

- 기본 필터 상태: `Docs`
- all-files 상태: `Files`

이유:

- 사용자는 지금 "문서 중심으로 보고 있다"는 점을 바로 이해할 수 있다.
- `a` 토글 후에는 "이제 전체 파일을 보고 있다"는 차이가 명확해진다.

## 단축키 평가표

### 유지해야 하는 핵심 키

| 키 | 범위 | 기능 정의 | 사용 방법 | 사용자 가치 | 판단 |
|---|---|---|---|---|---|
| `j` / `k`, `↑` / `↓` | browser, reader, full outline | 현재 컨텍스트 내 이동 | 파일 선택 이동 또는 문서 스크롤 | 가장 기본적인 탐색 수단 | 유지 |
| `Enter` | browser, full outline | 열기/진입/점프 | 폴더 진입, 문서 열기, outline 섹션 점프 | 학습 비용이 낮고 핵심 흐름과 일치 | 유지 |
| `h` / `←` / `Backspace` | browser, reader | 상위 이동 또는 닫기 | 폴더 상위 이동, reader 이탈 | 터미널 사용자에게 익숙함 | 유지 |
| `Esc` | reader, full outline | 현재 읽기 맥락 종료 | reader 종료, full outline 닫기 | 빠른 탈출 키로 유용함 | 유지 |
| `q` | 전역 | 앱 종료 | popup 닫기 | tmux popup 흐름과 직접 연결됨 | 유지 |

### 유지하되 문맥 제한이 필요한 키

| 키 | 범위 | 기능 정의 | 사용 방법 | 사용자 가치 | 문제 | 판단 |
|---|---|---|---|---|---|---|
| `/` | reader 전용 | 현재 문서 내 텍스트 검색 시작 | 검색어 입력 후 `Enter` | 긴 AI 문서 검토에 매우 유용 | browser에서 보이면 오해 발생 | 유지, reader에서만 노출 |
| `n` / `N` | reader 전용 | 다음/이전 검색 결과 이동 | `/` 이후 탐색 | 검색 기능의 완성도 유지 | 검색 전에는 의미 불분명 | 유지, 검색 상태에서만 강조 |
| `o` | reader 전용 | outline 보기 방식 전환 | full outline, side outline, 닫기 순환 | 구조가 긴 문서에 유용 | 단독으로 의미 추측이 어려움 | 유지, 설명 문구 개선 |
| `PgUp` / `PgDn`, `g` / `G` | reader, full outline | 빠른 상하 이동 | 페이지 단위 또는 처음/끝 이동 | 긴 문서에서 효율적 | 초보자에게는 보조 기능 | 유지, 보조 help로 이동 가능 |

### 기능은 유효하지만 이름/키/노출 방식이 불명확한 항목

| 키 | 범위 | 기능 정의 | 사용 방법 | 사용자 가치 | 문제 | 판단 |
|---|---|---|---|---|---|---|
| `a` | browser 전용 | 문서 중심 보기와 전체 파일 보기 전환 | 숨겨진 비문서 파일까지 보고 싶을 때 사용 | 특정 상황에서는 유용 | `a`만 보고는 의미를 알기 어려움 | 조건부 유지 |
| `[` / `]` | reader 전용 | 이전/다음 heading으로 점프 | 긴 문서에서 섹션 단위 이동 | outline 없이 빠르게 훑기 좋음 | `[/] heading`은 거의 해석 불가 | 조건부 유지 |
| `z` | reader 전용 | 좌측 패널 숨김/복구 | 문서에 집중할 때 사용 | 좁은 popup에서 유용 | 기본 컨셉 대비 빈도는 낮음 | 조건부 유지 |
| `r` | browser, reader | 수동 새로고침 | 자동 감지가 실패했을 때 fallback | 안전장치로는 유효 | 자동 리로드가 이미 있어 존재감이 약함 | 숨김 후보 |

## 현재 기준 권장 판단

### 반드시 남겨야 하는 것

- 파일/폴더 이동
- 문서 열기
- reader 종료
- 앱 종료
- 문서 내 검색

이 다섯 축이 `skimd`의 본질이다.

### 남기되 기본 help에서는 과감히 축소해야 하는 것

- `o`
- `PgUp` / `PgDn`
- `g` / `G`
- `a`
- `z`
- `r`
- `[` / `]`

기능 자체는 유효하지만, 첫 화면에서 모두 보여줄 필요는 없다.

### 제거까지 검토 가능한 것

- `r`
  - auto reload가 충분히 안정적이면 기본 사용자는 거의 안 쓴다.
- `z`
  - popup 크기가 충분하고 side outline이 passive라면 중요도가 떨어질 수 있다.

단, 둘 다 바로 제거할 대상은 아니다. 사용 로그가 없으므로 일단 "보조 기능으로 숨기는 것"이 우선이다.

## 추천 UX 방향

### 1. 도움말을 문맥별로 분리한다

browser에서 보여줄 것:

- `j/k` move
- `Enter` open
- `h` up
- `q` quit
- `a` all files

reader에서 보여줄 것:

- `j/k` scroll
- `Enter` open
- `/` search
- `o` outline
- `[` / `]` section
- `Esc` back
- `q` quit

full outline에서 보여줄 것:

- `j/k` move
- `Enter` jump
- `o` side
- `Esc` close

핵심은 "지금 눌러서 동작하는 키만 보이게 한다"는 것이다.

### 2. 도움말 문구를 동사 중심으로 바꾼다

좋지 않은 예:

- `a all`
- `[/] heading`
- `o outline`

좋은 방향:

- `a all files`
- `[/] prev/next section`
- `o outline view`

짧더라도 의미가 읽혀야 한다.

### 3. 패널 이름을 상태 설명형으로 바꾼다

권장:

- 기본 상태: `[ Docs ]`
- 전체 파일 상태: `[ Files ]`
- 문서 읽기: `[ Reading ]`
- 구조 보기: `[ Outline ]`

`Review`는 제품 설명에는 맞을 수 있지만 패널 레이블로는 맞지 않는다.

### 4. advanced shortcut는 `?` 또는 2단계 help로 내린다

한 줄 help bar에는 핵심 5~6개만 보여준다.

예시:

- browser: `j/k move  •  Enter open  •  h up  •  q quit`
- reader: `j/k scroll  •  / search  •  o outline  •  Esc back  •  q quit`

고급 기능은 `?`를 눌렀을 때 전체 목록으로 보여주는 방식이 더 적합하다.

## 구현 우선순위 제안

### 1단계: 용어와 help 문구 정리

- `Review` 제거
- `a`, `o`, `[` / `]` 설명 개선
- browser에서 `/` 제거

효과:

- 가장 작은 변경으로 가장 큰 혼란 감소

### 2단계: context-aware help bar

- browser / reader / full outline별 help 분리
- 실제 동작 가능한 키만 노출

효과:

- "보이는데 안 되는 키" 문제 해결

### 3단계: advanced shortcut 정리

- `a`, `z`, `r`, `[` / `]`의 기본 노출 여부 재검토
- 필요하면 `?` help screen 추가

효과:

- 처음 보는 사용자 기준 진입 장벽 감소

## 최종 권장안

현재 시점의 추천은 다음과 같다.

- 유지:
  - `j/k`, `Enter`, `h`, `Esc`, `q`, `/`, `n/N`, `o`
- 조건부 유지:
  - `a`, `[` / `]`, `z`, `PgUp/PgDn`, `g/G`
- 숨김 또는 제거 후보:
  - `r`

그리고 UI 용어는 아래로 정리하는 것이 가장 자연스럽다.

- `Review` -> `Docs`
- `All Files`는 유지
- help는 context-aware로 변경
